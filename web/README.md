# malcolmston/go — web

The landing site and search UI for the **go aggregator** (the Node.js ecosystem,
reimagined in Go). Built with **Next.js 15** (App Router) and **React 19**, styled
with Tailwind, and sharing its components with the per-library docs sites via the
local `go-ui` package (`../ui`).

The same codebase ships to two targets:

- **Vercel** — a full Next.js app. The API route handlers under `app/api/*`
  (GraphQL, search, health) run as serverless functions, backed by Elasticsearch
  when configured.
- **GitHub Pages** — a fully static export (`next build` with `output: 'export'`),
  served under the `/go` project-pages path. There are no server functions there,
  so the UI falls back to the bundled data files (`graph.json`,
  `search-index.json`, `docs/*.json`).

## Prerequisites

- Node.js 20+
- `npm install` (run once; installs this app plus the linked `go-ui` workspace)

Environment variables are documented in [`.env.example`](./.env.example). Copy it
to `.env.local` for local overrides — you rarely need any of them for plain
`next dev`.

## Develop

```bash
npm run dev        # next dev — http://localhost:3000
```

The dev server serves the app at the domain root (`NEXT_PUBLIC_BASE_PATH` blank)
with the `/api/*` route handlers live, so search and the package graph work
end-to-end locally. Without `ELASTICSEARCH_URL`/`ELASTICSEARCH_API_KEY` set,
search transparently falls back to an in-memory BM25 ranker.

## Build

```bash
npm run build      # next build (Vercel / full Next output, incl. API routes)
npm start          # serve the production build locally
```

### Static export (GitHub Pages)

```bash
GITHUB_PAGES=true NEXT_PUBLIC_BASE_PATH=/go npm run build
```

`GITHUB_PAGES=true` switches `next.config.mjs` to `output: 'export'` with
`basePath: '/go'`, emitting the static site into **`web/out`**.
`NEXT_PUBLIC_BASE_PATH=/go` makes client `fetch()`s and asset links resolve under
the `/go` prefix (via `withBase()` in `src/basePath.ts`). This export has no API
routes; the UI uses the bundled data files instead. This is exactly what the
`Pages` GitHub Actions workflow runs before uploading `web/out`.

## Routing

The App Router gives every tab a real path (replacing the old hash routing):

| Tab                 | Path              |
| ------------------- | ----------------- |
| Home                | `/`               |
| Parity              | `/parity`         |
| Pipeline            | `/pipeline`       |
| Explore             | `/explore`        |
| Releases            | `/releases`       |
| How-to              | `/howto`          |
| FAQ                 | `/faq`            |
| AI                  | `/ai`             |
| About               | `/about`          |
| A library (e.g. express) | `/lib/<id>`  |

The shared `go-ui` shell (nav, tab bar, footer) lives in `app/layout.tsx`, wired
to the Next router; each route's `page.tsx` renders its view client-only. The
route ↔ tab-id mapping is in `app/nav.ts`.

## API routes

Served by Next route handlers on the same host as the app (same-origin — there is
no separate API URL to configure):

| Route           | Method     | Purpose                                        |
| --------------- | ---------- | ---------------------------------------------- |
| `/api/graphql`  | POST / GET | GraphQL over the package graph + search index. |
| `/api/search`   | GET        | `?q=…&first=…` → ranked search hits + backend.  |
| `/api/health`   | GET        | `{ ok, es }` — liveness + Elasticsearch status. |

They share the query/ranking logic in `../api/_lib/*`.

### Elasticsearch (server-side)

`ELASTICSEARCH_URL` and `ELASTICSEARCH_API_KEY` enable the Elasticsearch-backed
search backend. They are **configured in the Vercel project environment**
(Project Settings → Environment Variables) and are **never committed** — not in
`.env.example`, not in the client bundle. When they are unset the API routes fall
back to an in-memory BM25 ranker, and on hosts with no server functions (GitHub
Pages) the frontend falls back to the bundled data files. See
[`.env.example`](./.env.example).

## Generated data

Several inputs are generated by the repository's build steps (see the root
`Pages` workflow) rather than hand-edited:

- `public/docs/<lib>.json` — per-library API reference (from `gendocs`).
- `src/parity.ts` — live upstream parity metrics (from `genparity`).
- `public/graph.json` + `public/search-index.json` — the package-connection graph
  and search index the Explore tab falls back to (`node scripts/build-graph-data.mjs`
  at the repo root, i.e. `npm run build:graph`).

## Layout

```
web/
  app/            Next App Router — layout, routes, and app/api/* route handlers
  src/
    components/   view components (Home, Parity, Pipeline, Explore, …)
    data/         library catalogue (LIBS)
    api/          client-side API helpers (graph.ts)
    basePath.ts   NEXT_PUBLIC_BASE_PATH helper (withBase)
  public/         static assets + generated data (docs, graph, search-index)
  tests/          unit + e2e tests
```
