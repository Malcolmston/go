Why this exists

🎯 Familiar APIs

If you know Express or Passport in Node, you already know these. The same concepts — handlers, middleware, strategies, rooms — mapped to idiomatic Go.

🪶 Almost no dependencies

Real protocols implemented from scratch on the standard library: RFC 6455 WebSockets, Engine.IO/Socket.IO codecs, JWT/JWKS, ANSI. Only chalk pulls in golang.org/x/term.

🔌 Interoperable

Socket.IO talks to the real socket.io-client@4. Passport JWTs verify against jsonwebtoken. Express mounts any net/http handler.

How to use this

Every library is an ordinary Go module. Install what you need with go get, import it, and go.

1 · Install

# pick any subset — they're independent modules
go get github.com/malcolmston/express
go get github.com/malcolmston/passport
go get github.com/malcolmston/socketio
go get github.com/malcolmston/chalk
go get github.com/malcolmston/morgan

2 · A minimal Express server

package main

import (
    "log"
    "github.com/malcolmston/express"
)

func main() {
    app := express.New()
    app.Get("/", func(req *express.Request, res *express.Response, next express.Next) {
        res.Send("Hello World")
    })
    log.Fatal(app.Listen(":3000"))
}

3 · The full stack together

Here Express serves the API, morgan logs every request, Passport guards a route, and Socket.IO handles realtime — all on one net/http server.

package main

import (
    "net/http"
    "github.com/malcolmston/express"
    "github.com/malcolmston/morgan"
    socketio "github.com/malcolmston/socketio"
)

func main() {
    // 1. HTTP routes with Express
    app := express.New()
    app.Get("/api/hello", func(req *express.Request, res *express.Response, next express.Next) {
        res.JSON(map[string]any{"msg": "hi"})
    })

    // 2. Realtime with Socket.IO
    io := socketio.New()
    io.OnConnection(func(s *socketio.Socket) {
        s.Join("general")
        s.On("chat", func(args []any) []any {
            io.To("general").Emit("chat", args...)
            return nil
        })
    })

    // 3. io.Handler intercepts /socket.io/ and delegates the rest to Express
    handler := io.Handler(app)

    // 4. Wrap everything in morgan request logging
    http.ListenAndServe(":3000", morgan.New(handler, morgan.Dev, morgan.Config{}))
}
Because every library is an http.Handler (or produces one), they compose the same way any Go middleware does — no framework lock-in.

Built with AI · Ready for AI

The AI story

This entire suite was designed, ported and tested with AI assistance — and it's built to be easy for AI coding agents to consume.

How it was built

Every library here was ported from its Node.js original with Claude (Anthropic) driving the implementation: reading the reference behaviour, writing idiomatic Go, generating table-driven tests, and verifying wire-level compatibility against the real Node clients. Large surfaces (100+ middleware packages, 100+ auth strategies) were produced by fanning out parallel agents, each self-verifying with gofmt, go vet and go test before integration.

🧭 Faithful behaviour

Ports were checked against the original semantics — e.g. encodeurl, ms, JWT signing, Socket.IO framing — not just "looks right".

✅ Verified, not vibes

Findings were adversarially re-checked and every package ships tests. CI runs build + vet + test on Go 1.23 and 1.24.

📚 Self-documenting

A dependency-free go/doc generator produces the API sites, so docs never drift from the code.

Using these libraries with AI agents

Because the APIs mirror the famous Node originals, LLMs already "know" the shape of them — which makes them unusually good targets for AI-assisted development:

  • Prompt with the Node name. "Add a Passport local strategy" or "broadcast to a Socket.IO room" map almost 1:1 to these APIs.
  • Point your agent at the docs. The GitHub Pages API reference for each repo is plain, crawlable HTML — ideal context for tools like Claude Code.
  • Small dependency graph. With almost no third-party packages, an agent can reason about the whole call graph without pulling in an ecosystem.
Transparency: AI wrote the code, humans set the direction, and the test suites + protocol interop checks are what make it trustworthy. Treat these as you would any dependency — read the docs, run the tests, pin your versions.

About

malcolmston/go is a family of Go libraries that recreate the most-loved building blocks of the Node.js ecosystem — with the same ergonomics, on top of Go's standard library.

Design principles

Familiar first

Mirror the original API and mental model so the learning curve is near zero for anyone coming from Node.

Standard library only

No framework, no sprawling dependency tree. Protocols are implemented from scratch so you can read and trust them.

Real interop

Where a wire format exists (Socket.IO, JWT, WebSocket), it's validated against the actual Node clients.

Tested & documented

Every package has tests and a generated API site; CI enforces fmt/vet/test across Go versions.
